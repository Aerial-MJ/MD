### 参数

- `n`：要生成的序列的数量，默认为 1。
- `best_of`：从多少个序列中选择最佳序列，需要大于 n，默认等于 n。
- `temperature`：用于控制生成结果的随机性，较低的温度会使生成结果更确定性，较高的温度会使生成结果更随机。
- `top_p`：用于过滤掉生成词汇表中概率低于给定阈值的词汇，控制随机性。
- `top_k`：选择前 k 个候选 token，控制多样性。
- `presence_penalty`：用于控制生成结果中特定词汇的出现频率。
- `frequency_penalty`：用于控制生成结果中词汇的频率分布。
- `repetition_penalty`：用于控制生成结果中的词汇重复程度。
- `use_beam_search`：是否使用束搜索来生成序列。
- `length_penalty`：用于控制生成结果的长度分布。
- `early_stopping`：是否在生成过程中提前停止。
- `stop`：要停止生成的词汇列表。
- `stop_token_ids`：要停止生成的词汇的ID列表。
- `include_stop_str_in_output`：是否在输出结果中包含停止字符串。
- `ignore_eos`：在生成过程中是否忽略结束符号。
- `max_tokens`：生成序列的最大长度。
- `logprobs`：用于记录生成过程的概率信息。
- `prompt_logprobs`：用于记录生成过程的概率信息，用于特定提示。
- `skip_special_tokens`：是否跳过特殊符号。
- `spaces_between_special_tokens`：是否在特殊符号之间添加空格。

这些参数的设置通常取决于具体需求和模型性能。以下是一些常见的设置指导方法：

- `temperature`：较低的温度（如0.2）会产生更确定性的结果，而较高的温度（如0.8）会产生更随机的结果。您可以根据您的需求进行调整。
- `presence_penalty、frequency_penalty 和 repetition_penalty`：这些参数可以用于控制生成结果中的词汇分布和重复程度。您可以根据您的需求进行调整。
- `use_beam_search`：束搜索通常用于生成更高质量的结果，但可能会降低生成速度。您可以根据您的需求进行调整。
- `length_penalty`：这个参数可以用于控制生成结果的长度。较高的值会产生更长的结果，而较低的值会产生更短的结果。您可以根据您的需求进行调整。
- `early_stopping`：如果您不希望生成过长的结果，可以设置此参数为True。
- `stop 和 stop_token_ids`：您可以使用这些参数来指定生成结果的结束条件。

## max_token 的两种含义

### vLLM 的两种 token 限制

| 参数 | 控制范围 | 在哪设置 |
|------|---------|---------|
| `max_model_len` | 输入 + 输出的总 token 数（上下文窗口） | 启动 server / 初始化 `LLM` 时 |
| `max_tokens` | 仅输出部分最多生成多少 token | 每次请求的 `SamplingParams` 中 |

```python
# max_model_len：模型级别，总上下文长度
llm = LLM(model=model_path, max_model_len=4096)

# max_tokens：请求级别，仅控制输出长度
sampling_params = SamplingParams(max_tokens=256)
```

### Transformers 的两种 max token

| 参数 | 控制范围 | 推荐 |
|------|---------|------|
| `max_length` | 输入 + 输出总长，超过截断 | ⚠️ 容易踩坑 |
| `max_new_tokens` | 仅新生成的 token 数，不管输入多长 | ✅ 推荐 |

```python
# 不推荐：输入400 token + max_length=512，输出只剩112 token
model.generate(input_ids, max_length=512)

# 推荐：不管输入多长，输出固定最多256 token
model.generate(input_ids, max_new_tokens=256)
```

> **常见坑**：用 `max_length` 时，若输入本身已很长，输出会被严重压缩，优先使用 `max_new_tokens`。

# vLLM PagedAttention 解析

## 原文中的问题

```
分块存储：PagedAttention 将每个序列的 Key 和 Value 缓存分为多个块（Blocks），每个块包含固定数量的 Tokens 的 Key 和 Value 张量。这种分块方式允许模型按需加载和处理部分数据，而不是整个序列的全部数据。

非连续显存空间利用：在操作系统中，虚拟内存允许程序使用连续的地址空间，而实际的物理内存可以是非连续的。Paged Attention 允许在非连续的显存空间中存储连续的 Key 和 Value 数据块。这样可以更有效地利用显存，尤其是在显存碎片化严重的情况下。

按需加载和计算：通过只加载和处理需要参与当前计算的 Key 和 Value 块，可以显著降低每次 Attention 操作的显存需求。这不仅降低了内存占用，还可以提高计算效率。

动态调整：根据实际需要，系统可以动态调整块的大小和数量，以优化性能和显存使用。
```


你的直觉是对的，原文有几处描述**不够准确甚至有误导性**，我来逐一指出：

---

### ❌ 问题1："按需加载部分数据，而不是整个序列的全部数据"

这个说法**容易产生误解**。

**实际情况是：**
- Forward 时，当前 step 的 **Query** 确实只有新生成的那1个 token（decode 阶段）
- 但 **Key/Value** 需要当前序列**所有历史 token** 的 KV cache 都参与 Attention 计算
- PagedAttention **并没有减少参与计算的 KV 数据量**，它只是改变了这些 KV 数据的**存储和寻址方式**

> 所以"按需加载部分数据"这个说法是**错的**。整个序列的 KV cache 都要参与计算，PagedAttention 解决的是**怎么存**的问题，而不是**存多少**的问题。

---

### ❌ 问题2："动态调整块的大小和数量"

Block size 在 vLLM 中是**启动时静态配置的超参数**（默认16），运行时**不会动态调整块大小**。动态的部分只是**块的分配数量**（按需申请新 block）。

---

### ✅ 说对的部分

- 分块存储 KV cache ✅
- 非连续物理显存存储逻辑上连续的数据 ✅（这个类比操作系统虚拟内存分页是核心思想）
- 降低显存碎片化 ✅

---

## PagedAttention 真正解决的问题

要理解它，先看**没有 PagedAttention 时的问题**：

```
传统方式：为每个请求预分配一块连续显存
┌─────────────────────────────────┐
│  Request A  │  Request B  │ ??? │
│  预分配2048  │  预分配2048  │碎片 │
└─────────────────────────────────┘
问题：
1. 不知道序列最终多长 → 要么预分配过多(浪费)，要么不够用
2. 不同请求的 KV cache 长度不一 → 大量内存碎片
3. 显存利用率只有 20%~40%
```

**PagedAttention 的解法：**

```
逻辑视图（程序看到的）：
Request A: [Block0] → [Block1] → [Block2]   (逻辑连续)

物理视图（显存实际布局）：
┌──────┬──────┬──────┬──────┬──────┬──────┐
│Block2│Block0│Block5│Block1│Block3│Block4│
│(A)   │(B)   │(A)   │(A)   │(B)   │(C)   │
└──────┴──────┴──────┴──────┴──────┴──────┘
         ↑物理不连续，但通过Block Table映射
```

**Block Table 是关键：**
```
Request A 的 Block Table:
逻辑Block 0 → 物理Block 1
逻辑Block 1 → 物理Block 3  
逻辑Block 2 → 物理Block 0
```

---

## 结合 HBM 和 SRAM 来讲

这里引入 GPU 存储层次结构：

```
GPU 存储层次（速度从慢到快，容量从大到小）：

┌─────────────────────────────────────────┐
│           CPU DRAM (主机内存)            │  ~TB级，很慢
└─────────────────────────────────────────┘
                    ↕ PCIe
┌─────────────────────────────────────────┐
│     HBM (High Bandwidth Memory)         │  
│     即"显存" ~40-80GB                   │  ~3 TB/s 带宽
│     存放：模型权重、KV cache、激活值     │
└─────────────────────────────────────────┘
                    ↕ 片上互联
┌─────────────────────────────────────────┐
│     SRAM (L1/L2 Cache / Shared Mem)     │  
│     片上缓存 ~几十MB                    │  ~19 TB/s 带宽
│     存放：当前计算的小块数据            │
└─────────────────────────────────────────┘
                    ↕
┌─────────────────────────────────────────┐
│           CUDA Cores / Tensor Cores     │  实际做矩阵乘法
└─────────────────────────────────────────┘
```

### Attention 计算中 HBM ↔ SRAM 的数据流动

**标准 Attention（无优化）的问题：**

```
每次 Attention 计算：

HBM → SRAM:  读取 Q, K, V          (大量 HBM 读)
SRAM → 计算: QK^T                   
SRAM → HBM:  写回 Attention Score   (HBM 写)
HBM → SRAM:  再读 Score             (HBM 读)
SRAM → 计算: Score × V
SRAM → HBM:  写回输出               (HBM 写)

问题：中间结果反复在 HBM 和 SRAM 之间搬运
      HBM 带宽成为瓶颈（Memory Bound）
```

**PagedAttention + FlashAttention 的优化：**

```
核心思想：把 K、V 分成小块(Tile)，
         每块都能装进 SRAM，在 SRAM 内完成计算

Step 1: Q 从 HBM 加载到 SRAM（只加载一次）
Step 2: 循环处理每个 KV Block：
        ┌──────────────────────────────────┐
        │  从 HBM 加载 KV Block_i 到 SRAM  │
        │  在 SRAM 内计算 Q × K_i^T        │
        │  在 SRAM 内做 Softmax（online）   │
        │  在 SRAM 内计算 × V_i            │
        │  累加到输出（不写回 HBM！）       │
        └──────────────────────────────────┘
Step 3: 最终输出写回 HBM（只写一次）
```

### PagedAttention 在这里扮演的角色

```
HBM 中的 KV Cache 布局（PagedAttention管理）：

物理Block 1: [token0~15 的 KV]  ← 地址 0x1000
物理Block 3: [token16~31 的 KV] ← 地址 0x5000  (不连续!)
物理Block 0: [token32~47 的 KV] ← 地址 0x0000  (不连续!)

计算时，按 Block Table 的映射：
依次把 Block1, Block3, Block0 从 HBM 搬到 SRAM
→ 在 SRAM 内完成该 block 的 Attention 计算
→ 累加结果
→ 下一个 block

关键：物理不连续没关系！
     每次只搬一个 block 到 SRAM，
     SRAM 只需要装得下一个 block 就够了
```

---

## 总结

| 概念 | 作用层面 | 解决的问题 |
|------|---------|-----------|
| **PagedAttention** | HBM 内部的 KV 管理 | 显存碎片、利用率低 |
| **FlashAttention** | HBM ↔ SRAM 的数据搬运 | HBM 带宽瓶颈 |
| **两者结合** | 端到端优化 | 既省显存又快 |

**一句话总结：**
- PagedAttention 管的是 KV cache 在 **HBM 里怎么摆放**（分页、非连续、按需分配 block）
- FlashAttention 管的是计算时 **HBM 和 SRAM 之间怎么搬数据**（分块计算、减少 HBM 读写次数）
- 两者是**互补**的，vLLM 同时使用了这两种技术

## 三、线程执行模型：从线程到SM的映射

```
软件视角（程序员写的）          硬件视角（GPU执行的）
                               
Grid（网格）                    
├── Block 0 ──────────────→   SM 0 执行
├── Block 1 ──────────────→   SM 1 执行  
├── Block 2 ──────────────→   SM 0 执行（复用）
├── Block 3 ──────────────→   SM 2 执行
│     ↓                            ↓
│   Thread 0~31 ─────────→   Warp 0（32线程 = 1个Warp）
│   Thread 32~63 ────────→   Warp 1
│   Thread 64~95 ────────→   Warp 2
│       ...                      ...
└── Block N ──────────────→   SM X 执行

关键规则：
• 1 个 Block 只能在 1 个 SM 上执行（不能跨SM）
• 1 个 SM 可以同时驻留多个 Block
• Warp 是调度的最小单位（32个线程）
• A100 每个SM最多驻留 64 个 Warp（2048线程）
```

### Warp 调度机制（隐藏延迟的关键）

```
时间轴：
        Cycle 1    Cycle 2    Cycle 3    Cycle 4    Cycle 5
        
Warp 0: [发射指令] [等待内存]  [等待内存]  [等待内存]  [收到数据]
Warp 1:            [发射指令] [等待计算]  [完成]      
Warp 2:                       [发射指令] [等待内存]  [等待内存]
Warp 3:                                  [发射指令]  [等待计算]

↑ 当 Warp 0 在等待内存时，调度器立刻切换到 Warp 1
  这就是 GPU 隐藏内存延迟的核心机制：用多 Warp 并发来填满流水线
  
前提：需要足够多的活跃 Warp → 这就是"Occupancy（占用率）"的意义
```

### SRAM vs HBM 的本质区别

```
SRAM（片上缓存/共享内存）：
┌─────────────────────────────────────────┐
│ • 物理上集成在 GPU 芯片内               │
│ • 速度极快，延迟~几十cycle              │
│ • 容量极小（A100全部SM加起来~20MB SRAM）│
│ • 程序员可以显式控制（共享内存）        │
│ • 每个SM私有，不同SM不能直接共享        │
└─────────────────────────────────────────┘

HBM（高带宽显存）：
┌─────────────────────────────────────────┐
│ • 物理上在 GPU 芯片旁边（通过硅中介层连接）│
│ • 速度较慢，延迟~几百cycle              │
│ • 容量大（40GB~80GB）                  │
│ • 存放模型权重、KV Cache、激活值等      │
│ • 所有 SM 共享访问                      │
└─────────────────────────────────────────┘
```

## 五、LLM 推理中的完整数据流

把上面所有概念串联起来：

```
LLM Decode 阶段（生成第 N+1 个 token）：

CPU/主机内存                HBM（显存）              SM内（SRAM+寄存器）
                                                    
                    ┌─── 模型权重（静态）              
                    │    ~几十GB                    
                    │                              
                    ├─── KV Cache（动态增长）         
                    │    PagedAttention 管理          
                    │    物理Block非连续存储           
                    │                              
                    └─── 当前输入 token embedding    
                                                    
                              ↓ 数据搬运到SM          
                    ┌─────────────────────────────┐  
                    │  SM 0                        │  
                    │  ┌──────────────────────┐   │  
                    │  │ 共享内存(SRAM)         │   │  
                    │  │ 装入 KV Block_i       │   │  
                    │  │ 装入 Q（当前token）    │   │  
                    │  │                      │   │  
                    │  │ Tensor Core 计算：    │   │  
                    │  │ Q × K^T → Score      │   │  
                    │  │ Softmax（online）     │   │  
                    │  │ Score × V → 累加输出  │   │  
                    │  └──────────────────────┘   │  
                    │  循环处理所有 KV Block        │  
                    └─────────────────────────────┘  
                              ↓ 写回结果              
                    ┌─────────────────────────────┐  
                    │  HBM：存储 Attention 输出    │  
                    │  → FFN 层计算               │  
                    │  → 下一层 Transformer       │  
                    └─────────────────────────────┘  
                              ↓ 最终输出              
              CPU ← 新生成的 token（通过 PCIe 传回）
```

### FlashAttention 在 SM 层面的作用

```
没有 FlashAttention：
HBM → SRAM: 读 Q,K,V
SRAM → HBM: 写中间 Score矩阵    ← 大量 HBM 读写
HBM → SRAM: 读 Score矩阵
SRAM → HBM: 写输出
问题：Score矩阵 = seq_len × seq_len，很大，HBM带宽打满

有 FlashAttention（Tiling 分块）：
将 K,V 切成小块，每块能装进 SRAM
循环：
  HBM → SRAM: 读一个 KV Block（小）
  SRAM 内部: 完成该块的 QK^T + Softmax + ×V
  结果在 SRAM 内累加（不写回HBM！）
最后：SRAM → HBM: 只写一次最终输出

减少 HBM 读写次数 → 从 Memory Bound 变为 Compute Bound
```

## 六、性能优化的核心矛盾

```
LLM 推理的两大瓶颈：

Prefill 阶段（处理输入 prompt）：
┌──────────────────────────────────────┐
│ • 大量 token 并行计算                 │
│ • Tensor Core 利用率高               │
│ • 瓶颈：Compute Bound（算力不够）     │
│ • SM 全部满载                        │
└──────────────────────────────────────┘

Decode 阶段（逐个生成 token）：
┌──────────────────────────────────────┐
│ • 每次只生成1个token，计算量极小      │
│ • 大量时间在读 KV Cache（HBM读取）   │
│ • 瓶颈：Memory Bound（带宽不够）     │
│ • SM 大量空闲，Tensor Core 利用率<10%│
└──────────────────────────────────────┘

对应的优化手段：
Decode Memory Bound → Continuous Batching（多请求共享SM）
                    → PagedAttention（提高KV Cache利用率）
                    → 量化（减少HBM读取数据量）
                    → Speculative Decoding（增加计算密度）
```

# vLLM Block Size 详解

## Block Size 是什么

```
Block Size = 每个物理Block能存放的 Token 数量

默认 Block Size = 16，意味着：
每个物理Block 可以存放 16 个 token 的 KV Cache
```

---

## 一个 Block 里存的是什么

```
单个 Token 的 KV Cache 结构：

每一层 Transformer Layer 都有：
┌─────────────────────────────────────────┐
│  Key   向量: [d_head × num_heads] 维度  │
│  Value 向量: [d_head × num_heads] 维度  │
└─────────────────────────────────────────┘

所以一个 Block（Block Size=16）存放的是：
┌──────────────────────────────────────────────────────┐
│  Block 内容（以单层为例）                              │
│                                                      │
│  token_0:  K[0], V[0]                               │
│  token_1:  K[1], V[1]                               │
│  token_2:  K[2], V[2]                               │
│  ...                                                 │
│  token_15: K[15], V[15]                             │
│                                                      │
│  共 16 个 token 的 KV 数据                            │
└──────────────────────────────────────────────────────┘

注意：实际上每一层都有独立的 Block
     一个请求在每一层都有自己的 Block Table
```

---

## Block 如何随序列增长动态分配

```
以 Block Size = 4 为例（方便演示）：

━━━━ 时刻1：输入 prompt = "你好，今天" （4个token）━━━━

申请 1 个 Block：
┌─────────────────┐
│ Block #5        │  ← 从空闲池分配
│ [你][好][，][今] │  → 4个token，Block 刚好填满
└─────────────────┘

Block Table:
逻辑Block 0 → 物理Block #5


━━━━ 时刻2：生成第5个token "天" ━━━━

Block #5 已满！申请新 Block：
┌─────────────────┐    ┌─────────────────┐
│ Block #5        │    │ Block #12       │  ← 新分配
│ [你][好][，][今] │    │ [天][  ][  ][  ]│  → 只用了1格
└─────────────────┘    └─────────────────┘

Block Table:
逻辑Block 0 → 物理Block #5
逻辑Block 1 → 物理Block #12


━━━━ 时刻3：继续生成 "气" "真" "好" ━━━━

┌─────────────────┐    ┌─────────────────┐
│ Block #5        │    │ Block #12       │
│ [你][好][，][今] │    │ [天][气][真][好]│  → Block填满
└─────────────────┘    └─────────────────┘


━━━━ 时刻4：继续生成 "啊" ━━━━

又满了，再申请：
┌──────────┐    ┌──────────┐    ┌──────────┐
│ Block #5 │    │ Block #12│    │ Block #3 │ ← 新分配
│[你好，今] │    │[天气真好] │    │[啊][ ][ ][ ]│
└──────────┘    └──────────┘    └──────────┘
```

---

## Block Size 大小的影响

```
Block Size 太小（比如 = 1）：
┌──────────────────────────────────────────┐
│ 优点：内存浪费极少（最后一个Block几乎满）  │
│ 缺点：Block Table 很长，管理开销大        │
│       每次 Attention 要跳转很多次         │
│       不利于 GPU 的连续内存访问           │
└──────────────────────────────────────────┘

Block Size 太大（比如 = 256）：
┌──────────────────────────────────────────┐
│ 优点：Block Table 短，管理简单            │
│       连续内存访问，对 GPU 友好           │
│ 缺点：最后一个 Block 浪费严重             │
│                                          │
│  例：序列长度 = 257                       │
│  需要 2 个 Block，第二个 Block 只用 1 格  │
│  浪费 255/512 = 约 50% 的显存！           │
└──────────────────────────────────────────┘

Block Size = 16（vLLM 默认）：
┌──────────────────────────────────────────┐
│ 平衡点：                                  │
│ 平均浪费 = Block Size / 2 = 8 个 token   │
│ 管理开销适中                              │
│ GPU 访问效率较好                          │
└──────────────────────────────────────────┘
```

---

## 内存浪费的量化理解

```
对于任意一个请求，最多浪费多少显存？

最坏情况：序列长度 = N × BlockSize + 1
          最后一个 Block 只用了 1 个槽位
          浪费了 (BlockSize - 1) 个槽位

平均浪费：BlockSize / 2 个 token 的 KV Cache

BlockSize=16 时：
平均每个请求浪费 8 个 token 的 KV Cache
= 8 × 2(K+V) × num_heads × d_head × num_layers × sizeof(dtype)

对于 LLaMA-7B（FP16）：
= 8 × 2 × 32 × 128 × 32 × 2 bytes
= 8 × 524288 bytes
≈ 4 MB 的浪费（可接受）
```

## 和操作系统分页的类比

```
操作系统虚拟内存           vLLM PagedAttention
─────────────────────────────────────────────
页（Page）          ←→    Block
页大小（4KB）        ←→    Block Size（16 tokens）
物理页帧            ←→    物理 Block
页表（Page Table）  ←→    Block Table
缺页中断            ←→    按需分配新 Block
内存碎片减少        ←→    显存碎片减少

核心思想完全一致：
逻辑上连续 → 物理上可以不连续
通过"表"来做映射
按需分配，不预留
```

## 总结一句话

```
Block Size = 16 的含义：

显存被切成一个个"格子"
每个格子能装 16 个 token 的 KV Cache
序列每增长 16 个 token，就申请一个新格子
格子在显存里可以不连续，通过 Block Table 找到它们

就像停车场：
  不是给每辆车预留一整块固定区域
  而是来一辆分一个车位
  车位编号记在停车单上
  车位可以分散在停车场各处
```
