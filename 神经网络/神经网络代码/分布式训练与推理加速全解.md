## 一、先建立层次概念：这些框架不在同一层

初学时最容易困惑的地方是：DP、DDP、FSDP、DeepSpeed、Megatron、Accelerate、TRL、verl 听起来都是"分布式训练相关"，但它们其实分属四个不同的层次，互相不是竞争关系，而是**层层叠加**的关系：

```
┌─────────────────────────────────────────────────────┐
│  第4层：你写的训练代码（应用层）                       │
└─────────────────────────────────────────────────────┘
                          ↓ 调用
┌─────────────────────────────────────────────────────┐
│  第3层：训练工具库（封装层）                           │
│   Accelerate / TRL / verl / OpenRLHF / LLaMA-Factory  │
└─────────────────────────────────────────────────────┘
                          ↓ 底层依赖
┌─────────────────────────────────────────────────────┐
│  第2层：分布式训练引擎（实现层）                       │
│   DeepSpeed / Megatron-LM / PyTorch FSDP              │
└─────────────────────────────────────────────────────┘
                          ↓ 具体实现
┌─────────────────────────────────────────────────────┐
│  第1层：分布式策略（思想层）                           │
│   DP / DDP / ZeRO / Tensor Parallel / Pipeline Parallel│
└─────────────────────────────────────────────────────┘
                          ↓ 最终依赖
┌─────────────────────────────────────────────────────┐
│                  PyTorch / CUDA / NCCL                │
└─────────────────────────────────────────────────────┘
```

一句话记忆：

```
DP/DDP/FSDP/TP/PP  →  分布式的"方法论"（怎么切）
DeepSpeed/Megatron  →  具体实现这些方法的"引擎"
Accelerate          →  让你的代码兼容多种引擎的"适配器"
TRL/verl/OpenRLHF   →  在引擎上封装 RLHF/SFT 逻辑的"工具库"
```

## 二、第 1 层：分布式策略（思想层）

### 2.1 DP（Data Parallel，数据并行）

最简单的并行方式：把数据切分给多张卡，每张卡都持有**完整的模型副本**，各自计算梯度后再汇总更新。

```
┌──────────────────────────────────────────┐
│  数据: [batch1, batch2, batch3, batch4]  │
│         ↓       ↓       ↓       ↓        │
│       GPU0    GPU1    GPU2    GPU3       │
│      [模型]  [模型]  [模型]  [模型]      │
│         ↓       ↓       ↓       ↓        │
│       梯度1   梯度2   梯度3   梯度4      │
│         └───────┴───────┴───────┘        │
│                   ↓                      │
│              汇总梯度（AllReduce）        │
│                   ↓                      │
│              更新所有卡的模型             │
└──────────────────────────────────────────┘
```

PyTorch 里的 `nn.DataParallel` 是**单进程多线程**实现，主卡承担梯度汇总和参数分发的工作，受 Python GIL 限制，多卡扩展性差，现在基本已被 DDP 取代。

**问题**：每张卡都要存完整模型，模型太大时单卡装不下，DP 就失效了。

### 2.2 DDP（Distributed Data Parallel）

DP 的升级版，PyTorch 原生支持，是目前单机/多机多卡训练的标准选择。

| 对比项 | DP | DDP |
|--------|----|----|
| 进程模型 | 单进程多线程（受 GIL 限制） | 多进程（每个 GPU 一个独立进程） |
| 通信方式 | 等所有梯度算完再汇总 | 边计算边通信（梯度计算与 AllReduce 通信 overlap） |
| 扩展性 | 差，多卡加速比低 | 好，接近线性加速 |

```
┌─────────────────────────────────┐
│  进程0(GPU0)    进程1(GPU1)      │
│  [完整模型]     [完整模型]       │
│       ↓               ↓         │
│    计算梯度         计算梯度      │
│       └───AllReduce───┘         │
│       ↓               ↓         │
│    同步更新         同步更新      │
└─────────────────────────────────┘
```

```python
import torch.distributed as dist
from torch.nn.parallel import DistributedDataParallel as DDP

dist.init_process_group(backend="nccl")
model = model.to(local_rank)
model = DDP(model, device_ids=[local_rank])

for input_ids, labels in loader:
    result = model(input_ids, labels=labels)
    loss = result.loss
    loss.backward()          # backward 内部自动完成梯度 AllReduce
    optimizer.step()
    optimizer.zero_grad()
```

**限制**：每张卡仍需存完整的参数 + 梯度 + 优化器状态，适合"模型能放进单卡显存"的场景；一旦模型规模超过单卡显存，DDP 本身无法解决，需要引入下面的显存分片方案。

### 2.3 FSDP / ZeRO（显存分片，DP 的显存优化版）

**核心问题**：DDP 中每张卡都存一份完整的参数、梯度、优化器状态，这是严重的冗余。

**核心思想**：把参数、梯度、优化器状态**切碎分散**到各张卡上，需要计算时再临时通信拼出完整数据，用完立刻释放。

```
┌──────────────────────────────────────────┐
│  DDP（每卡存完整模型）：                   │
│  GPU0: [全部参数][全部梯度][全部优化器]    │
│  GPU1: [全部参数][全部梯度][全部优化器]    │
│  GPU2: [全部参数][全部梯度][全部优化器]    │
│  → 严重冗余！                             │
│                                           │
│  FSDP/ZeRO（切片存储）：                  │
│  GPU0: [参数1/3][梯度1/3][优化器1/3]      │
│  GPU1: [参数2/3][梯度2/3][优化器2/3]      │
│  GPU2: [参数3/3][梯度3/3][优化器3/3]      │
│  → 显存大幅减少！                         │
│                                           │
│  需要计算时：临时 AllGather 拼完整参数     │
│  计算完立刻释放                           │
└──────────────────────────────────────────┘
```

- **ZeRO** 是 DeepSpeed（微软）提出的思想，**FSDP（Fully Sharded Data Parallel）是 PyTorch 官方原生实现，思想上等价于 ZeRO-3**。
- 优点：可以训练远超单卡显存的大模型；缺点：通信开销比纯 DDP 更大。
- ZeRO 三个阶段的具体切分方式和显存节省的精确计算，见 [四、ZeRO 三阶段深度详解](#四zero-三阶段深度详解显存到底怎么切的)。

### 2.4 Tensor Parallel（张量并行，TP）

把模型**权重矩阵本身**切开，多张卡各算一部分，最后合并结果。

```
┌─────────────────────────────────┐
│  一个 Linear 层 W [4096 × 4096] │
│                                 │
│  GPU0: W 的左半部分 [4096×2048] │
│  GPU1: W 的右半部分 [4096×2048] │
│                                 │
│  计算时各自算一半，再合并结果    │
└─────────────────────────────────┘
```

适合场景：单层矩阵本身就大到一张卡放不下（或算力不够）；是 Megatron-LM 的核心技术。关于 MLP 层具体是"先列切、后行切"、以及为什么这样切通信量最小（GELU 非线性约束），已在 [MiniMind从0到1构建大模型 · 4.5.4 张量并行](MiniMind从0到1构建大模型.md#454-张量并行mlp-层如何切分到多卡) 中详细推导，此处不再重复。

### 2.5 Pipeline Parallel（流水线并行，PP）

把模型**按层切开**，不同卡负责不同的连续层段，数据像流水线一样依次流过各卡。

```
┌─────────────────────────────────┐
│  GPU0: Layer 1-8   （前8层）    │
│  GPU1: Layer 9-16  （中8层）    │
│  GPU2: Layer 17-24 （后8层）    │
│                                 │
│  数据像流水线一样流过各卡        │
└─────────────────────────────────┘
```

**核心问题：Bubble（气泡/GPU 空转）**。因为各阶段必须串行等待上一阶段的输出，朴素实现会导致大量 GPU 时间浪费在等待上。这是流水线并行最大的痛点，详见 [五、流水线并行的 Bubble 问题详解](#五流水线并行的-bubble-问题详解)。

### 2.6 Sequence Parallel（序列并行，SP，补充）

`test.txt` 的层次图中提到了 SP 但未展开，这里补充：序列并行是 Megatron-LM 后续引入的优化，把 **LayerNorm、Dropout 等激活值沿序列维度切分**到各张卡（这些操作在 batch/seq 维度上是逐元素独立的，不需要像 Attention/MLP 那样做张量并行的矩阵切分），进一步降低张量并行区域内的激活值显存占用，通常和 TP 配合使用（合称 TP+SP）。

## 三、第 2 层：分布式训练引擎（实现层）

### 3.1 DeepSpeed（微软）

核心贡献是 **ZeRO（Zero Redundancy Optimizer）**，此外还提供：

- **ZeRO-Offload**：把参数/优化器状态卸载到 CPU/NVMe，让单卡也能训练远超显存容量的模型
- 混合精度训练（FP16/BF16）
- 梯度累积、激活检查点（Activation Checkpointing）

使用方式是写一个 `ds_config.json` 配置文件，几乎不改动训练代码：

```json
{
  "train_micro_batch_size_per_gpu": 8,
  "gradient_accumulation_steps": 4,
  "bf16": {"enabled": true},
  "zero_optimization": {
    "stage": 2,
    "allgather_bucket_size": 2e8,
    "reduce_bucket_size": 2e8
  }
}
```

```python
import deepspeed

model_engine, optimizer, loader, scheduler = deepspeed.initialize(
    model=model, optimizer=optimizer, training_data=dataset, config="ds_config.json"
)

for batch in loader:
    loss = model_engine(batch).loss
    model_engine.backward(loss)   # 梯度分片同步在内部自动处理
    model_engine.step()
```

### 3.2 Megatron-LM（英伟达）

核心贡献是 **3D 并行**：Tensor Parallel + Pipeline Parallel + Data Parallel 三种并行方式组合使用，可以训练千亿/万亿参数级别的模型。

```
┌──────────────────────────────────────────┐
│           Megatron 3D 并行                │
│                                           │
│  Tensor Parallel  → 切权重矩阵            │
│  Pipeline Parallel → 切模型层             │
│  Data Parallel    → 切数据                │
│                                           │
│  三种并行组合使用                          │
│  可以训练万亿参数模型                      │
└──────────────────────────────────────────┘
```

特点：专为超大模型设计（GPT-3 级别）；与 NVIDIA 硬件深度绑定；代码侵入性强（需要改模型代码，把标准 `nn.Linear` 替换成 Megatron 提供的 `ColumnParallelLinear`/`RowParallelLinear` 等）；性能极致但使用门槛高。

### 3.3 DeepSpeed vs Megatron

| | DeepSpeed | Megatron-LM |
|---|-----------|--------------|
| 核心技术 | ZeRO 显存优化 | 3D 并行 |
| 使用难度 | 中等 | 高 |
| 代码侵入性 | 低 | 高 |
| 适合规模 | 10B 以下更常见 | 100B+ |
| 典型用户 | 普通研究者、中小团队 | 大厂预训练基础模型 |
| 开发者 | 微软 | 英伟达 |

现实中很多大厂采用 **Megatron-DeepSpeed** 结合方案：3D 并行负责切分模型和数据，ZeRO 负责进一步压缩每张卡上的显存占用，两者并不冲突，可以叠加使用（比如在 DP 维度上应用 ZeRO）。

### 3.4 PyTorch FSDP

PyTorch 官方原生实现，思想上是 ZeRO-3 的原生复刻（参数、梯度、优化器状态都切分），不需要额外安装 DeepSpeed，和 PyTorch 生态（`torch.compile` 等）集成更紧密，是目前 HuggingFace Accelerate/TRL 默认推荐的多卡方案之一。

## 四、ZeRO 三阶段深度详解：显存到底怎么切的

### 4.1 先回答两个常见疑问

**Q1：ZeRO 的切分是在 DP 基础上做的吗？**

是的。ZeRO 本质上**仍然是数据并行（DP）**——每张卡处理不同的数据分片，模型的计算逻辑不变。ZeRO 改变的只是"参数、梯度、优化器状态"这几类模型状态在多卡间**如何存储**，而不是引入新的并行计算方式。

**Q2：单卡也能用 DeepSpeed，那切分还有意义吗？**

单卡上 ZeRO 的"切分到多卡"没有意义（因为只有 1 张卡）。单卡使用 DeepSpeed 主要是为了用它的其他能力：混合精度训练、梯度累积、激活检查点，以及最重要的 **ZeRO-Offload**（把优化器状态/参数卸载到 CPU 内存甚至 NVMe 硬盘，用较慢的存储换取能训练更大的模型）。

### 4.2 训练时显存里到底存了什么

以参数量为 $\Psi$ 的模型、混合精度训练（FP16/BF16 计算 + FP32 优化器状态）、Adam 优化器为例：

| 显存占用项 | 大小（bytes） | 说明 |
|-----------|--------------|------|
| 模型参数（FP16/BF16） | $2\Psi$ | 前向/反向计算使用 |
| 梯度（FP16/BF16） | $2\Psi$ | 反向传播产生 |
| 优化器状态（FP32）— 最大头 | $12\Psi$ | FP32 主参数 $4\Psi$ + 一阶动量 $4\Psi$ + 二阶动量 $4\Psi$ |
| **合计（不含激活值）** | **$16\Psi$** | 7B 模型 → 约 112GB，单张 A100(80GB) 装不下 |

> 这里的精确数值和更细的正反梯度峰值分析（比如是否需要额外的 FP32 梯度副本导致 $18\Psi/20\Psi$）见 [LLM 大模型训练-推理显存占用分析](../../论文/docs/大模型/LLM%20大模型训练-推理显存占用分析.md)，本节只关注 ZeRO 切分的相对节省比例。

普通 DP（未使用 ZeRO）下，**N 张卡每张都存一份完整的 $16\Psi$**，总显存是 $16\Psi \times N$，其中有效信息只有 $16\Psi$，冗余比例是 $(N-1)/N$。

### 4.3 ZeRO-1：只切优化器状态

把 $12\Psi$ 的优化器状态平均切成 $N$ 份，每张卡只存 $1/N$：

```
每卡显存 = 2Ψ（完整参数） + 2Ψ（完整梯度） + 12Ψ/N（优化器状态分片）
```

以 $N=4$ 为例：每卡显存 $= 2\Psi + 2\Psi + 3\Psi = 7\Psi$（相比 DP 的 $16\Psi$，节省约 2.3×）。

训练流程：
```
前向传播：各卡用自己完整的参数正常计算（参数还是完整的，没切）
反向传播：各卡算出完整梯度，AllReduce 同步梯度（和普通 DP 一样）
参数更新：← 这里不一样了！
  每张卡只用自己存的那 1/N 优化器状态，更新对应的 1/N 参数
  更新完后，AllGather 把各卡更新的参数分片广播给所有人
  → 所有卡重新拥有完整参数
```

### 4.4 ZeRO-2：切优化器状态 + 梯度

在 ZeRO-1 基础上，梯度也切分：

```
每卡显存 = 2Ψ（完整参数） + 2Ψ/N（梯度分片） + 12Ψ/N（优化器状态分片）
```

以 $N=4$ 为例：每卡显存 $= 2\Psi + 0.5\Psi + 3\Psi = 5.5\Psi$（相比 DP 节省约 2.9×）。

训练流程的关键变化：反向传播不再用 AllReduce 同步完整梯度，而是用 **Reduce-Scatter**（求和后按分片切给各卡，每卡只保留自己负责的那一段梯度求和结果，其余立刻释放）。

**为什么用 Reduce-Scatter 而不是 AllReduce？**

```
AllReduce = Reduce-Scatter + AllGather

普通 DP： Reduce-Scatter（汇总梯度） + AllGather（广播回完整梯度）
         = 每张卡都有完整梯度

ZeRO-2：  只做 Reduce-Scatter 这一步！
         每卡只保留自己负责的梯度分片
         省掉了 AllGather 这一步 → 梯度显存直接减少 N 倍
```

### 4.5 ZeRO-3（≈ FSDP）：切优化器状态 + 梯度 + 参数

连参数也切分，三类状态全部按 $1/N$ 分片存储：

```
每卡显存 = 2Ψ/N（参数分片） + 2Ψ/N（梯度分片） + 12Ψ/N（优化器状态分片）
        = 16Ψ / N
```

以 $N=4$ 为例：每卡显存 $= 16\Psi/4 = 4\Psi$（相比 DP 节省 4×）；$N=64$ 时理论可降到约 $0.25\Psi$（节省 64×）——**这也是为什么会看到"ZeRO-3 理论节省 64×"这种说法：它对应的是 $N=64$ 张卡这个特定场景，倍数会随卡数 $N$ 线性提升，并不是一个固定值**，套用时要注意带上具体的 $N$。

训练流程最复杂，因为每一层计算时都需要该层的**完整**参数：

```
前向传播：
  计算 Layer k 之前：AllGather 出 Layer k 的完整参数
  所有卡用完整参数计算 Layer k
  计算完立刻丢掉其他卡的参数分片！只保留自己负责的 1/N
  继续下一层，重复上述过程

反向传播：
  同样需要 AllGather 参数（算梯度需要参数）
  算完梯度后 Reduce-Scatter，每卡只保留自己负责的梯度分片

参数更新：
  各卡独立更新自己负责的参数分片，不需要 AllGather
  （下一次前向时再按需重新 AllGather）
```

### 4.6 三阶段完整对比

| | 优化器状态 | 梯度 | 参数 | 通信操作 | 每卡显存（$N$ 卡） |
|---|-----------|------|------|---------|-------------------|
| 普通 DP | 完整副本 | 完整副本 | 完整副本 | AllReduce | $16\Psi$ |
| ZeRO-1 | 切分 $1/N$ | 完整副本 | 完整副本 | AllReduce + AllGather | $4\Psi + 12\Psi/N$ |
| ZeRO-2 | 切分 $1/N$ | 切分 $1/N$ | 完整副本 | Reduce-Scatter + AllGather | $2\Psi + 14\Psi/N$ |
| ZeRO-3（≈FSDP） | 切分 $1/N$ | 切分 $1/N$ | 切分 $1/N$ | AllGather（按需）+ Reduce-Scatter | $16\Psi/N$ |

> 表中公式是显存占用的通用表达式（$N$ 是任意卡数），代入 $N=4$ 就是本节推导的 $7\Psi/5.5\Psi/4\Psi$。切勿把某个具体 $N$ 下算出的倍数（如"4×/8×/64×"）当成与卡数无关的固定值。

### 4.7 ZeRO-Offload：为什么单卡也能用 DeepSpeed

单卡用 ZeRO 没有"切分给多卡"的意义，但可以用 **ZeRO-Offload**：把显存压力转移到更大但更慢的存储介质（CPU 内存或 NVMe 硬盘）。

```
┌─────────────────────────────────────────┐
│  GPU 显存（快但小）：                     │
│    参数（FP16）  ← 计算时在 GPU          │
│    梯度（FP16）  ← 计算时在 GPU          │
│                                          │
│  CPU 内存（慢但大）：                     │
│    优化器状态（FP32） ← 卸载到 CPU       │
│    FP32 主参数        ← 卸载到 CPU       │
│                                          │
│  流程：GPU 算梯度 → 传给 CPU → CPU 更新参数 → 传回 GPU 继续训练 │
└─────────────────────────────────────────┘
```

效果：单卡 24GB 显存也可能训练原本需要 80GB 显存的模型；代价是 CPU-GPU 之间的数据传输会引入额外延迟，训练速度会变慢。

## 五、流水线并行的 Bubble 问题详解

### 5.1 Bubble 是怎么产生的

流水线并行把模型按层切给多张卡，数据必须按顺序依次流过：GPU0 算完 → 传给 GPU1 → GPU1 算完 → 传给 GPU2 ...

```
前向传播阶段（朴素流水线，只有 1 个大 batch）：

时间轴 →→→→→→→→→→→→→→→→→→→→→→→→

GPU0: [算batch] [空闲] [空闲] [空闲]
GPU1: [空闲]    [算batch] [空闲] [空闲]
GPU2: [空闲]    [空闲]  [算batch] [空闲]
GPU3: [空闲]    [空闲]  [空闲]  [算batch]

GPU0 算完就闲置了，要等 GPU1/2/3 依次算完
→ 大量 GPU 时间浪费在等待（Bubble）上
```

加上反向传播后 Bubble 问题更严重：

```
Bubble 比例 = (GPU数 - 1) / GPU数

4 张卡 → Bubble 占比 = 3/4 = 75%！
→ 相当于 4 张卡里有 3 张在大部分时间空转
```

### 5.2 Micro-batch 如何缓解 Bubble（GPipe 方式）

**核心思想**：与其等一个大 batch 全部算完，不如把它切成很多个小的 micro-batch，让流水线上的每个阶段持续有活干，类似工厂流水线不等一辆车完全组装好才开始下一辆。

```
把 1 个 batch 切成 4 个 micro-batch（m1, m2, m3, m4）：

时间轴 →→→→→→→→→→→→→→→→→→→→→→→→→→→→→→→→

GPU0: [F,m1][F,m2][F,m3][F,m4]    [B,m4][B,m3][B,m2][B,m1]
GPU1:      [F,m1][F,m2][F,m3][F,m4][B,m4][B,m3][B,m2][B,m1]
GPU2:           [F,m1][F,m2][F,m3][F,m4][B,m4][B,m3][B,m2][B,m1]
GPU3:                [F,m1][F,m2][F,m3][F,m4][B,m4][B,m3][B,m2][B,m1]

Bubble 只在流水线的开头和结尾（图中留白部分），中间阶段各卡持续工作
```

**Bubble 比例公式**：

$$\text{Bubble 比例} = \frac{p-1}{m+p-1}$$

其中 $p$ 是流水线阶段数（GPU 数），$m$ 是 micro-batch 数量。

| $p$（GPU数） | $m$（micro-batch数） | Bubble 比例 | 说明 |
|---|---|---|---|
| 4 | 1（无 micro-batch） | $3/4 = 75\%$ | 很差 |
| 4 | 4 | $3/7 \approx 43\%$ | 好一些 |
| 4 | 8 | $3/11 \approx 27\%$ | 更好 |
| 4 | $\to \infty$ | $\to 0\%$ | 理想情况 |

结论：micro-batch 数量越多，Bubble 越少；但这引出了下一个问题——显存开销。

### 5.3 GPipe 的新问题：显存随 micro-batch 数线性增长

GPipe 虽然用 micro-batch 减少了 Bubble，但每张卡要把 **所有 micro-batch 的前向激活值都保存在显存里**，等反向传播时才用得上：

```
GPU0: [F,m1][F,m2][F,m3][F,m4]...

GPU0 要把 m1,m2,m3,m4 的激活值全部保存在显存里，等反向传播时用！
→ 显存占用 = micro-batch 数量 × 单个激活值大小
→ micro-batch 越多，显存压力越大
→ 和"减少 Bubble 需要更多 micro-batch"这个目标直接矛盾！
```

### 5.4 Megatron 的解决方案：1F1B（One Forward One Backward）

**核心思路**：不等所有 micro-batch 的前向全部算完再统一做反向，而是**算完一个 micro-batch 的前向就立刻做它对应的反向**，从而及时释放这个 micro-batch 占用的激活值显存。

```
时间轴 →→→→→→→→→→→→→→→→→→→→→→→→→→→→→→

GPU0: [F1][F2][F3][F4][B4][B3][B2][B1]
GPU1:    [F1][F2][F3][F4][B4][B3][B2][B1]
GPU2:       [F1][F2][F3][F4][B4][B3][B2][B1]
GPU3:          [F1][F2][F3][F4][B4][B3][B2][B1]

稳定阶段：每个 GPU 交替做前向和反向
  → 显存占用恒定（只需同时保存 p 个 micro-batch 的激活值，而不是 m 个）
  → Bubble 比例和 GPipe 相同（公式不变）
  → 但显存大幅减少！
```

> 注：不同资料对 1F1B 稳定阶段的具体调度时序描述略有差异（比如各卡进入"1 前向 1 反向"交替模式的起始时机不同），但核心结论一致：显存占用从 $O(m)$（GPipe，随 micro-batch 数增长）降低到 $O(p)$（1F1B，只随流水线阶段数增长），这是 1F1B 相对 GPipe 的核心改进。

### 5.5 Megatron 的进阶方案：Interleaved 1F1B

普通 1F1B 中每张卡负责**连续**的若干层：

```
GPU0: [Layer 1-8]
GPU1: [Layer 9-16]
```

Interleaved（交错）1F1B 让每张卡负责**不连续的多个层段**：

```
GPU0: [Layer 1-4]  + [Layer 17-20]
GPU1: [Layer 5-8]  + [Layer 21-24]
GPU2: [Layer 9-12] + [Layer 25-28]
GPU3: [Layer 13-16]+ [Layer 29-32]
```

效果：Bubble 比例可以进一步降低（大致按每卡负责的段数 $v$ 缩减），代价是通信量相应增加（每卡需要在更多层边界处收发激活值/梯度）。

### 5.6 完整方案对比

| 方案 | Bubble | 显存 | 复杂度 |
|------|--------|------|--------|
| 朴素流水线（无 micro-batch） | 75% | 低 | 简单 |
| GPipe（micro-batch） | 显著降低 | 高（随 $m$ 增长） | 中等 |
| 1F1B | 与 GPipe 相同 | 低（恒定为 $O(p)$） | 中等 |
| Interleaved 1F1B | 更低 | 低 | 复杂（通信量增加） |

**核心矛盾与结论**：Bubble 减少需要更多 micro-batch，而显存节省需要更少 micro-batch；1F1B 通过"算完一个前向立刻做对应反向"打破了这个矛盾，在不增加 Bubble 的前提下把显存占用降了下来，是目前较好的平衡方案。

## 六、通信原语：AllReduce、Scatter、Reduce-Scatter 到底有什么区别

### 6.1 完整的集合通信操作列表

```
Broadcast        一对多（复制同一份数据给所有人）
Scatter          一对多（切片分发，各人拿到不同的一片）
Gather           多对一（收集，Scatter 的反向操作）
Reduce           多对一（带计算，如求和/求平均）
AllGather        多对多（每人收集到所有人的完整数据）
AllReduce        多对多（带计算，结果同步给所有人）
Reduce-Scatter   多对多（先 Reduce 再 Scatter）
AllToAll         多对多（转置式交换，每人给每人发一份不同的数据）
```

### 6.2 逐个图解

**Broadcast（广播）**：一个节点把完整数据发给所有人。

```
GPU0: [A]  ──→  GPU0: [A]
GPU1: [ ]  ──→  GPU1: [A]
GPU2: [ ]  ──→  GPU2: [A]
```
场景：把模型参数从主节点广播给所有卡（初始化时同步权重）。

**Scatter（散播）**：一个节点把数据**切片**分发给所有人，每人拿到不同的一片。

```
GPU0: [A,B,C,D]  ──→  GPU0: [A]
                       GPU1: [B]
                       GPU2: [C]
                       GPU3: [D]
```
场景：把数据集切片分发给各卡。

**Gather（收集）**：Scatter 的反向操作，所有节点的数据汇集到一个节点。

```
GPU0: [A]  ┐
GPU1: [B]  ├──→  GPU0: [A,B,C,D]
GPU2: [C]  ┤
GPU3: [D]  ┘
```

**Reduce（规约）**：所有节点数据汇集到一个节点，并做计算（求和/求平均）。

```
GPU0: [1]  ┐
GPU1: [2]  ├──→  GPU0: [1+2+3+4=10]
GPU2: [3]  ┤
GPU3: [4]  ┘
```
场景：把所有卡的 loss 汇总到主卡打印。

**AllGather（全收集）**：每个节点都收集所有人的数据，最终每个节点都拥有完整数据（Gather 的"All"版本）。

```
GPU0: [A]  ──→  GPU0: [A,B,C,D]
GPU1: [B]  ──→  GPU1: [A,B,C,D]
GPU2: [C]  ──→  GPU2: [A,B,C,D]
GPU3: [D]  ──→  GPU3: [A,B,C,D]
```
场景：FSDP/ZeRO-3 前向计算时，每张卡只存 1/N 参数，计算前 AllGather 拼出完整参数。

**AllReduce（全规约）**：所有节点数据做计算，结果同步给所有人（本质上 = Reduce + Broadcast 两步合一）。

```
GPU0: [1]  ──→  GPU0: [10]
GPU1: [2]  ──→  GPU1: [10]
GPU2: [3]  ──→  GPU2: [10]
GPU3: [4]  ──→  GPU3: [10]
```
场景：DDP 梯度同步，每张卡算完梯度后 AllReduce 让所有卡的梯度保持一致。

**Reduce-Scatter（规约散播）**：先对所有数据做 Reduce（求和），再把求和结果按分片 Scatter 给各节点，每个节点只得到结果的一部分。

```
输入：
  GPU0: [a0, a1, a2, a3]
  GPU1: [b0, b1, b2, b3]
  GPU2: [c0, c1, c2, c3]
  GPU3: [d0, d1, d2, d3]

输出（每人只得到一段的求和结果）：
  GPU0: [a0+b0+c0+d0]   ← 第0段的总和
  GPU1: [a1+b1+c1+d1]   ← 第1段的总和
  GPU2: [a2+b2+c2+d2]   ← 第2段的总和
  GPU3: [a3+b3+c3+d3]   ← 第3段的总和
```
场景：ZeRO-2/3、FSDP 梯度同步，每张卡只负责保存和更新一部分梯度。

### 6.3 最重要的关系：AllReduce = Reduce-Scatter + AllGather

这是理解现代分布式训练通信优化的关键：

```
第一步 Reduce-Scatter：
  GPU0: [g0,g1,g2,g3]       GPU0: [G0]  ← 只有第0段的汇总
  GPU1: [g0,g1,g2,g3]  ──→  GPU1: [G1]  ← 只有第1段的汇总
  GPU2: [g0,g1,g2,g3]       GPU2: [G2]
  GPU3: [g0,g1,g2,g3]       GPU3: [G3]

第二步 AllGather：
  GPU0: [G0]       GPU0: [G0,G1,G2,G3]  ← 完整梯度
  GPU1: [G1]  ──→  GPU1: [G0,G1,G2,G3]
  GPU2: [G2]       GPU2: [G0,G1,G2,G3]
  GPU3: [G3]       GPU3: [G0,G1,G2,G3]

结果和直接 AllReduce 完全一样，但拆成两步后：
  ZeRO-2/3 可以在 Reduce-Scatter 之后就停止，不再做 AllGather
  → 每张卡只保留自己负责的梯度分片，省下了后一半的显存和通信
```

### 6.4 在训练框架中的实际应用

```
┌──────────────────────────────────────────────────┐
│  DDP 梯度同步：                                   │
│  各卡梯度 → AllReduce → 所有卡梯度一致            │
├──────────────────────────────────────────────────┤
│  FSDP / ZeRO-3 前向传播：                        │
│  各卡只有 1/N 参数 → AllGather → 完整参数         │
│  计算完立刻释放（节省显存）                        │
├──────────────────────────────────────────────────┤
│  FSDP / ZeRO-2/3 反向传播：                      │
│  各卡完整梯度 → Reduce-Scatter → 各卡 1/N 梯度    │
│  每张卡只更新自己负责的那部分参数                  │
├──────────────────────────────────────────────────┤
│  Tensor Parallel：                               │
│  各卡算一半结果 → AllReduce → 合并完整结果        │
└──────────────────────────────────────────────────┘
```

### 6.5 一张表总结 + 记忆口诀

| 操作 | 输入 | 输出 | 典型场景 |
|------|------|------|---------|
| Broadcast | 1 个节点有数据 | 所有人都拿到相同的完整数据 | 参数初始化广播 |
| Scatter | 1 个节点有完整数据 | 各人拿到不同的 1 片 | 数据集切片分发 |
| Gather | 各人各有 1 片 | 1 个节点拿到完整拼接结果 | 结果汇总 |
| Reduce | 各人各有数据 | 1 个节点拿到求和结果 | loss 汇总 |
| AllGather | 各人各有 1 片 | 所有人都拿到完整拼接结果 | FSDP 前向拼参数 |
| AllReduce | 各人各有数据 | 所有人都拿到求和结果 | DDP 梯度同步 |
| Reduce-Scatter | 各人各有数据 | 各人拿到自己那 1 段的求和结果 | FSDP/ZeRO 梯度同步 |

```
记忆口诀：

Scatter   → 一个人把东西分给大家（切片发出去）
Gather    → 大家把东西交给一个人（收集起来）
Reduce    → 大家把东西交给一个人，路上顺便做加法
All 前缀  → 不是只交给一个人，而是所有人最终都得到相同的结果

AllGather      = 大家都得到完整数据（人手一份拼好的）
AllReduce      = 大家都得到求和结果（人手一份算好的）
Reduce-Scatter = 先求和，再切片，每人只拿自己那一段
```

> 补充校正：口语中容易把 AllGather 说成"All-scatter"，两者方向相反——Scatter 是把完整数据"发出去"（切片），Gather/AllGather 是把切片数据"收回来"（拼完整），注意不要混淆。

## 八、推理侧：Attention 的几种实现方式（eager / SDPA / FlashAttention / PagedAttention）

训练侧关注"怎么把模型/数据切到多卡上"，推理侧还有一组同样重要但完全不同维度的优化：**同一个 Attention 数学公式，用不同的底层实现方式计算，速度和显存可以差几倍到十几倍**。这里的几种实现不是互相替代的并行策略，而是**同一个算子的不同实现后端（kernel）**。

### 8.1 Attention 的标准数学公式（先对齐基准）

$$\text{Attention}(Q,K,V) = \text{softmax}\left(\frac{QK^T}{\sqrt{d_k}}\right)V$$

不管用哪种实现方式，算出来的**数值结果都应该一致**（或在数值精度误差范围内一致），区别只在于**怎么算更快、更省显存**。

### 8.2 eager（朴素实现）

`eager` 是 HuggingFace Transformers 里 `attn_implementation="eager"` 对应的实现方式：直接按 PyTorch 算子逐步计算——`Q @ K.transpose(-1,-2)` → 除以 `sqrt(d_k)` → `softmax` → `@ V`，也就是本笔记 [手写Attention代码合集](基础代码/手写Attention代码合集.md) 中最基础的写法。

**问题**：中间会显式生成一个 `[seq_len, seq_len]` 的完整注意力分数矩阵（Attention Score Matrix），这个矩阵需要写入并从显存（HBM）中反复读写：

```
标准 Attention（eager）的问题：

每次 Attention 计算：
HBM → SRAM:  读取 Q, K, V              (大量 HBM 读)
SRAM → 计算: QK^T
SRAM → HBM:  写回 Attention Score 矩阵  (HBM 写，seq_len × seq_len 很大！)
HBM → SRAM:  再读 Score 矩阵            (HBM 读)
SRAM → 计算: Score × V
SRAM → HBM:  写回输出                  (HBM 写)

问题：中间结果反复在 HBM 和 SRAM 之间搬运
     序列越长，Score 矩阵越大（随 seq_len 平方增长）
     HBM 带宽成为瓶颈（Memory Bound，不是算力不够，是数据搬运太慢）
```

### 8.3 SDPA（`torch.nn.functional.scaled_dot_product_attention`）

**SDPA 不是某一种具体的加速算法，而是 PyTorch 官方提供的一个统一算子接口**，它会根据运行环境（硬件、精度、是否有 mask、序列长度等）自动在多种底层 kernel 之间选择最优实现，包括：

- **Math 后端**：等价于 eager 的朴素实现（数学上最直接，作为 fallback）
- **FlashAttention 后端**：满足条件时自动调用 FlashAttention 算法（见下）
- **Memory-Efficient Attention 后端**：类似 xFormers 提供的显存优化实现

```python
import torch.nn.functional as F

# 函数签名（PyTorch 官方）
output = F.scaled_dot_product_attention(
    query, key, value,
    attn_mask=None,
    dropout_p=0.0,
    is_causal=False,   # 因果 mask，用于自回归生成
    scale=None,        # 默认使用 1/sqrt(d_k)
    enable_gqa=False,  # 是否启用 Grouped Query Attention 加速路径
)
```

一句话理解：**SDPA 是接口，FlashAttention 是这个接口背后可能被调用的具体实现之一**。在 HuggingFace Transformers 里设置 `attn_implementation="sdpa"`，就是把 Attention 计算交给 PyTorch 自动选择最优后端，通常比 `eager` 快且省显存，且不需要额外安装第三方库。

### 8.4 FlashAttention（真正的算法创新）

FlashAttention 是 Stanford 团队提出的 Attention 精确计算加速算法（不是近似算法，数学结果完全等价），核心思路是用 **Tiling（分块）+ Online Softmax** 避免把完整的 Score 矩阵写入 HBM：

```
有 FlashAttention（Tiling 分块）：

核心思想：把 K、V 切成小块（Tile），每块都能装进 SRAM，在 SRAM 内完成计算

Step 1: Q 从 HBM 加载到 SRAM（只加载一次）
Step 2: 循环处理每个 KV Block：
        ┌──────────────────────────────────┐
        │  从 HBM 加载 KV Block_i 到 SRAM  │
        │  在 SRAM 内计算 Q × K_i^T        │
        │  在 SRAM 内做 Softmax（online）   │  ← 关键技巧：不需要提前知道全局最大值
        │  在 SRAM 内计算 × V_i            │
        │  累加到输出（不写回 HBM！）       │
        └──────────────────────────────────┘
Step 3: 最终输出写回 HBM（只写一次）

减少 HBM 读写次数 → 从 Memory Bound 变为 Compute Bound（真正发挥出 GPU 算力）
```

**Online Softmax** 是 FlashAttention 的关键数学技巧：标准 Softmax 需要先看到一整行的所有值才能算出最大值用于数值稳定（减去最大值防止溢出），FlashAttention 通过维护一个运行中的最大值和归一化因子，分块处理时增量更新之前的结果，从而不需要把整行数据都装进内存就能算出正确的 Softmax。

**性能数据**（来自 FlashAttention / FlashAttention-2 论文）：
- FlashAttention 在 A100 上相比标准实现有 **2~4 倍**的速度提升，序列越长优势越明显
- 由于不存储完整的中间注意力矩阵，显存占用可降低 **10~20 倍**
- FlashAttention-2 在 FlashAttention 基础上进一步提速 **2~3 倍**（主要通过减少非矩阵乘法运算、更好的并行划分实现）

### 8.5 eager / SDPA / FlashAttention 三者关系梳理

```
关系不是并列的三选一，而是"接口"与"实现"的分层关系：

┌─────────────────────────────────────────────┐
│  你的代码：model(input, attn_implementation=?)│
└─────────────────────────────────────────────┘
              ↓ 选择实现方式
   ┌──────────┼──────────────┐
   ↓          ↓              ↓
"eager"    "sdpa"      "flash_attention_2"
朴素实现   PyTorch统一接口   直接指定用 FlashAttention 库
   │          │              │
   │    自动选择最优后端：     │
   │    ├─ Math（≈eager）    │
   │    ├─ FlashAttention ───┤ ← 两条路径最终都可能落到同一个 kernel
   │    └─ Memory-Efficient  │
   ↓          ↓              ↓
结果数值一致，速度/显存差异巨大
```

在 HuggingFace Transformers 中，`attn_implementation` 参数可以显式指定为 `"eager"`、`"sdpa"` 或 `"flash_attention_2"`（后者需要额外 `pip install flash-attn`）；不指定时库会自动探测环境并选择可用的最优实现。

### 8.6 PagedAttention（vLLM，解决的是另一个维度的问题）

**容易混淆的点**：PagedAttention 和 FlashAttention 优化的**不是同一个环节**，两者是互补关系，不是竞争关系。

| 概念 | 作用层面 | 解决的问题 |
|------|---------|-----------|
| **FlashAttention** | 单次 Attention 计算内部，HBM ↔ SRAM 的数据搬运 | 计算时的显存带宽瓶颈（Memory Bound） |
| **PagedAttention** | 多个请求的 KV Cache 在 HBM 里怎么存放 | 显存碎片化、利用率低（推理服务场景） |

PagedAttention 借鉴操作系统虚拟内存分页的思想：把每个请求的 KV Cache 切成固定大小的 Block（默认 16 个 token 一块），Block 在物理显存中可以不连续存放，通过一张 Block Table 做逻辑地址到物理地址的映射：

```
逻辑视图（程序看到的）：
Request A: [Block0] → [Block1] → [Block2]   (逻辑连续)

物理视图（显存实际布局）：
┌──────┬──────┬──────┬──────┬──────┬──────┐
│Block2│Block0│Block5│Block1│Block3│Block4│
│(A)   │(B)   │(A)   │(A)   │(B)   │(C)   │
└──────┴──────┴──────┴──────┴──────┴──────┘
         ↑ 物理不连续，但通过 Block Table 找到它们
```

**需要澄清的两个常见误解**（vLLM 论文/资料中容易被误读的地方）：

1. ❌ "PagedAttention 按需加载部分 KV 数据参与计算" —— **不准确**。Decode 阶段虽然 Query 只有新生成的 1 个 token，但 Key/Value 仍然需要当前序列**全部历史 token** 的 KV Cache 参与计算，参与计算的数据量并没有减少。PagedAttention 解决的是这些 KV 数据"**怎么存**"（分页、非连续、按需分配 block），而不是"**存/算多少**"的问题。
2. ❌ "运行时动态调整 block 大小" —— **不准确**。Block Size 是启动时的静态配置超参数（vLLM 默认 16），运行期间不会改变；动态的部分只是**block 的分配数量**会随序列增长按需申请。

两者结合时的完整数据流：vLLM 按 Block Table 把物理上分散的 KV Block 依次从 HBM 搬到 SRAM，每次只搬一个 block（FlashAttention 式的分块思想），在 SRAM 内完成该 block 的 Attention 计算并累加结果，兼顾了"显存高效存储"（PagedAttention）和"计算高效搬运"（FlashAttention）两个维度。更完整的 GPU 架构（SM/HBM/SRAM/Warp 调度）视角，见 [VLLM.md · GPU 架构全景](../HuaggingFace/VLLM.md)。

### 8.7 GQA 与 Attention 实现方式的关系

[手写Attention代码合集 · MultiQueryAttention](基础代码/手写Attention代码合集.md#三multiqueryattention-mqa) 中介绍的 MQA/GQA 是**模型结构层面**的设计（减少 K/V 头数），而这一节讨论的 eager/SDPA/FlashAttention 是**同一个 Attention 结构的不同计算实现**，两者是正交的优化维度，可以叠加：比如一个用了 GQA 结构的模型，推理时依然可以选择用 SDPA 或 FlashAttention 后端来加速计算。

需要注意的是，PyTorch SDPA 对 GQA 的支持是有条件的：目前只在 CUDA 上的 FlashAttention 和 Math 后端支持 `enable_gqa=True`，且要求 Query 头数能被 Key/Value 头数整除。

### 8.8 一张表总结推理加速的几个层次

| 层次 | 涉及技术 | 解决的问题 |
|------|---------|-----------|
| 模型结构层 | MHA → GQA → MQA → MLA | 减少 K/V 头数或做低秩压缩，从根本上减小 KV Cache 体积 |
| 单次计算实现层 | eager → SDPA → FlashAttention | 减少 HBM 读写次数，把 Memory Bound 变成 Compute Bound |
| 多请求显存管理层 | PagedAttention | 解决多个并发请求的 KV Cache 碎片化、利用率低的问题 |
| 服务调度层 | Continuous Batching、Chunked Prefill | 提高 GPU 利用率，让 Prefill/Decode 请求高效混合执行 |

> 服务调度层的 Continuous Batching、Chunked Prefill 等内容详见 [vLLM参数与推理机制详解.md](vLLM参数与推理机制详解.md)。
